01设计目标与分层原则

code_v2 作为产品化版本，目标不是继续堆叠 QSDemo 适配代码，而是形成长期可维护、可替换芯片 SDK、可复用量产业务能力的设备端工程结构。三套代码定位明确，互不混淆。

  • 媒体链恢复、低功耗切换
  • 录像 / RTSP / OSD 真实调用顺序
  • 只参考行为，不照搬单文件结构

现有设备端工程经验（分层参考）

  • 接口命名方式
  • 宿主侧 / 板端双态适配思路

  • 保留不删，但不再继续膨胀

code_v2（正式主线）

  • 新功能、新接口、新目录优先落这里
  • 先冻结接口/事件/状态/返回码
  • 再逐步映射 QSDemo 真实实现
  • 本文档优先级高于个人临时写法

app · 业务层

04依赖方向与实现禁区

依赖方向单向向下，严禁反向依赖。这是整个架构能否长期维持的关键约束。

允许的主要依赖方向

# 配置流
products        ──▶ app/config

# 业务编排
app/core        ──▶ app/{cloud,config,preview,record,alarm}
app/core        ──▶ libsys / libmedia

# 业务子域
app/cloud       ──▶ libsys/modem
app/cloud/ota   ──▶ app/cloud + products/升级包策略
app/preview     ──▶ libmedia 对外媒体接口
app/storage     ──▶ libmedia 对外媒体接口 + app/storage/dhfs
app/alarm       ──▶ libmedia/detect + app/cloud + app/storage

# 能力层
libmedia        ──▶ libmedia/adapter/ax615
libsys/*        ──▶ 各自实现层 + AX/SoC SDK

禁止反向依赖

  • libmedia 不能依赖 app/*
  • libsys 不能依赖 app/*
  • libmedia 不能直接依赖 app/cloud
  • libsys/modem 不能解释云协议业务含义
  • app/* 不能直接 include AX SDK 头文件

默认实现禁区

  • 在 app/* 中直接调用 AX SDK
  • 在 libmedia 中写 AOV 业务状态机
  • 在 libsys/modem 中写云协议业务语义
  • 在 app/cloud 中直接拉 GPIO 或访问 4G 驱动
  • 在多个模块重复维护工作模式/电量阈值/录像路径/唤醒原因
  • 没有接口边界就直接从业务层跳到底层实现
单一真值来源：工作模式、电量阈值、录像路径、唤醒原因字段、板端路径常量等，只能在一处定义，禁止散落多个目录重复维护。低功耗录像规格只由 periodic_wakeup_interval_sec 单一配置项驱动。

05核心数据模型与接口

所有跨层事件以"事实"形式上送，决策只在 app/core 产生。下列类型与接口对应实际代码中的 aov_types.hpp / domain_types.hpp / media_types.hpp / sys_types.hpp 及各 i_*.hpp 接口头。三层各自维护独立的命名空间（aov::app / aov::media / aov::sys）与状态码，跨层只通过下列稳定结构体传值，不暴露各自内部对象。

核心枚举与数据结构 · app/core/aov_types.hpp

enum class ConfiguredWorkMode {   // 用户可配置模式
    PowerSaving,                 // 省电
    SmartNoSleep,                // 智能不休眠
    UltraLongStandby,            // 超长待机
};

enum class RuntimeWorkState {     // 运行时有效状态
    AwakeNormal,                 // 常电正常
    SleepPeriodicDetect,         // 定时唤醒检测
    SleepAppWakeOnly,            // 仅 App 唤醒
    LowBatteryProtection,        // 极低电量保护
};

enum class WakeupSource {
    Timer, Detect, AppRemote, Manual, Unknown,
};
struct WorkModePolicy {
    ConfiguredWorkMode configured_mode {PowerSaving};
    int periodic_wakeup_interval_sec {1};   // 1/3/5/15 单一真值
    int normal_record_fps {15};
    int low_battery_protect_threshold_pct {10}; // 5~30
    int smart_to_save_threshold_pct {70};
    int smart_to_awake_threshold_pct {80};
};

struct BatteryStatus {
    int percent {100}; bool charging {false};
    bool low_battery_protect {false};
};

struct DetectResultSummary {
    bool has_person, has_pet, has_vehicle;
    int object_count; uint64_t frame_id, timestamp_ms;
};

跨层交互数据契约（app ↔ libmedia / app ↔ libsys）

下列结构是业务层与能力层之间的稳定传值边界：app 不感知 AX SDK 对象，libmedia / libsys 也不感知云协议语义。每层各自维护独立状态码（AovStatusCode / MediaStatusCode / SysStatusCode），跨层调用只在接口边界做一次映射，SDK 原生错误码不向上穿透。

跨层映射约定：同名概念在不同层用各自命名空间表达，仅在接口边界转换一次。例如 libmedia 的 MediaStatusCode::Busy 进入 app/core 时映射为 AovStatusCode::Busy；libsys 的唤醒/电量事实回到 core 后才参与 ResolveRuntimeState() 决策。app 永远拿不到、也不应拿到 AX SDK 句柄或 4G 驱动对象。

06工作模式状态机

app/core 同时维护"配置模式"（用户设置）与"运行时有效状态"（实际行为），二者不混为一个字段。运行时状态由配置模式 + 电池状态 + 预览状态共同决议（见 ResolveRuntimeState()）。

配置模式 → 运行时状态映射

配置模式条件运行时状态
PowerSaving正常SleepPeriodicDetect
SmartNoSleep电量 ≥ 70%AwakeNormal
电量 < 70%SleepPeriodicDetect（降级）
UltraLongStandby—SleepAppWakeOnly
任意模式电量 ≤ 保护阈值LowBatteryProtection（覆盖一切）

电池阈值规则

  • smart_to_save_threshold_pct = 70（降级到省电）
  • smart_to_awake_threshold_pct = 80（恢复常电）
  • low_battery_protect_threshold_pct = 5~30，默认 10
  • 退出保护：battery_pct > 阈值 + 10

各模式能力约束

省电模式

  • 定时唤醒间隔 1 / 3 / 5 / 15 秒
  • 录像间隔 = 唤醒间隔 = 低功耗取帧间隔（如 1秒1帧）
  • 检测到目标 → 恢复常电 + 切 15fps 正常录像
  • 无事件 → 回休眠，等待下次唤醒

智能不休眠 / 超长待机

  • 智能不休眠：默认常电，按 70/80 阈值降级与恢复
  • 超长待机：不自动唤醒，仅支持 App 远程唤醒查看

极低电量保护（运行时保护态）

  • 不自动定时唤醒；仅允许 App 唤醒查看
  • 禁用云台；禁止一键报警；关闭补光灯
  • 能力裁剪必须落到接口行为

命名约定：每个能力 = 一个 i_xxx.hpp（纯虚接口）+ 一个 xxx_stub.{hpp,cpp}（可编译占位实现）。后续接 AX615/QSDemo 真实实现时，只替换 stub，接口与上层调用不变。遵循"多个小文件优于少数大文件"。
Packet 当前策略：app/packet 承接 libmedia 输出的裸编码帧，不再加私有帧头/尾，也不在下游做剥头动作。Packet 统一承担缓存、跨线程分发、引用计数和背压控制；SD 卡链路在 app/storage/dhfs 内做 PS/PES 封装，云端和预览链路直接消费裸帧或交给云 SDK 封装。
Storage 迁移策略：app/storage 当前先建立 PacketFrame -> StorageService -> DhfsWriter 主链，参考 Apps/Kylin/comps/Storage/PSEncapsulation.* 迁移 PS/PES 封装；Depend/third_party/include/DHFS 与 Depend/third_party/lib/libDHFS.a 提供 DHFS 文件系统接口。下一任务补齐真实 PS/PES 封装与 DHFS 写盘，把 DhfsWriterStub 替换为真实 writer。PSAnalysis.* 属于回放/下载解析，SDStoragePublic.* 属于旧产品 SD 策略与公共工具，先记录为后续迁移项，本阶段不搬。
AOV 回休眠策略：所有业务链路进入终态后，app/core 不立即休眠，而是启动“空闲防抖等待（3-10 秒）”。等待期间若 App 预览、告警、云控或录像再次触发，则取消本次休眠；等待到期后再次确认本地录像已“关闭文件 + 刷缓存 + 强制落盘”、云存已“成功或明确失败”，再由 libsys/power 执行 AOV 休眠。

09事件 / 回调 / 状态码规范

统一跨模块的事件表达、回调线程边界与返回码，避免每个模块各发明一套。

事件模型

  • 四类：电源/系统 · 媒体检测 · 云/App · 会话/录像
  • 统一 EventHeader{timestamp_ms, sequence}
  • 命名用完成式：WakeupTriggeredEvent、DetectCompletedEvent
  • 事件只携带事实，不携带"应该怎么办"
  • 决策只放 app/core

回调与线程模型

  • 底层只"通知事件发生"，不做重业务
  • 统一 EventCallback<T> = function<void(const T&)>
  • 媒体线程禁止直接做云交互/复杂决策
  • 跨模块事件汇聚到 core 单线程消费
  • 禁止回调里阻塞网络 / 重磁盘 IO / 递归切状态

状态码规范

  • 全局 StatusCode：Ok / InvalidArgument / InvalidState / Busy / Timeout / Unsupported / NoResource…
  • 启停/配置/状态切换类不只返回 bool
  • 需保留底层细节用 ErrorInfo{code, vendor_code, message}
  • SDK 原生错误码不穿透到业务层
  • 日志用于诊断，不能解析日志决定业务流程

10计划实现顺序

按“先主链、再复用、后细化”的顺序推进。优先打通 AOV 休眠 / 唤醒、检测、录像、预览、上报闭环；可复用能力尽量先通过适配层接入，减少重复开发。

① 状态机与配置主链

  • app/core + app/config
  • 工作模式、电量门限、定时唤醒间隔
  • 先固化配置生效链与运行态决策
  • AOV 休眠期间允许 App/云端修改参数，但只写入云端目标配置 DesiredConfig；设备实际生效配置 ActiveConfig 等设备下次唤醒、同步、校验并应用成功后更新
  • 普通参数不为立即生效单独唤醒设备；检测开关、检测类型、灵敏度、AOV 唤醒间隔、事件录像时长等在下次 PIR/定时/远程唤醒后同步并生效
  • 配置状态需可追踪：已同步、待生效、已生效、校验失败、需要重启、需要唤醒执行

② 系统输入与唤醒链路

  • libsys/power + libsys/modem + libsys/device
  • 定时唤醒、远程唤醒 GPIO、4G 在线与心跳
  • 优先复用现有 4G 模块能力，补齐心跳适配

③ 媒体生产与 Packet 主链

  • libmedia + app/packet
  • libmedia 负责低功耗取帧、常电编码、检测、抓图、OSD、媒体链启停
  • app/packet 负责裸编码帧承载、缓存、引用计数、跨线程分发、背压控制
  • 主链重点：libmedia -> app/packet，这里不承载预览和录像业务决策
app/packet 开发阶段：
  1. 阶段 1：建立 IPacketService、PacketFrame、MediaPacketRouter，先用内存版实现接收 libmedia 裸码流并分发给 SD 和云端。
  2. 阶段 2：填充产品化 Packet 依赖，接入 Depend/third_party/include 与 Depend/third_party/lib/libPacket.a，替换底层内存分配和引用计数。
  3. 阶段 3：对接 libmedia::StreamFrameCallback，把 VENC/JENC/Audio 裸帧统一发布到 Packet，不加私有帧头。
  4. 阶段 4：接入 app/storage 和 app/cloud 两类当前消费者，补齐背压、丢帧策略、统计和退出 drain；app/preview 仅作为预留消费者。

④ 本地存储链路

  • app/storage + app/storage/dhfs + libsys/SD
  • SD 卡录像：Packet 裸帧 -> PS/PES 封装 -> 写 SD 卡文件 -> 关闭文件 + 刷缓存 + 强制落盘
  • app/storage 负责本地录像生命周期、空间策略、目录策略和写盘调度
  • app/storage/dhfs 负责 PS/PES 封装和 SD 卡文件写入
  • 近期迁移：PSEncapsulation.*；后续迁移：PSAnalysis.* 用于回放/下载，SDStoragePublic.* 用于旧产品 SD 策略参考
  • libsys/SD 只负责 SD 检测、在线状态、挂载、卸载、上电/断电等驱动层能力
  • SD 卡低功耗策略：上电 / 挂载 / 保持挂载 / 写盘 / 卸载 / 断电，避免事件录像期间频繁挂载卸载

⑤ 云端交互与云存链路

  • app/cloud + app/alarm + app/preview
  • 云端预览：Packet 裸帧 -> 云 SDK 封装 -> 云端 -> 手机 App 预览
  • 云存：Packet 裸帧 -> 云存 SDK 封装 -> 云端存储 -> 成功或明确失败后回 app/core
  • 告警：抓图 / 录像索引 / 告警消息通过 app/cloud 上报云端
  • 同时承接配置下发、状态回传、直播回放和控制命令

可复用模块

  • 4G 模块：远程唤醒触发、在线检测、心跳适配
  • 腾讯云配置接口：配置下发、状态同步、命令回执
  • 文件系统：区分常电模式与 AOV 模式下的存储差异
  • 云台算法：沿用现有控制与跟踪相关能力
  • Flash：参数持久化、工厂配置、运行配置存储

落地注意点

  • 挂载点、目录、空间检查与 SD 卡写盘策略统一收口到 app/storage
  • 码流文件外层私有协议统一收口到 app/storage/dhfs，不下沉到 libmedia
  • 媒体、4G、云端三条链先打通最小闭环，再补充云台、补光灯、抓图优化
  • 芯片 SDK、4G 模组、云协议的替换范围只允许收口在适配层
  • 分辨率、码率、帧率、编码格式、OSD、音频编码等媒体链参数，在恢复常电、启动预览或启动录像前应用；如果当前媒体链已运行，需按“停止链路 → 应用配置 → 重启链路”处理
  • 实时预览、远程唤醒、云台控制、手动抓图、手动录像、格式化 SD 卡、重启和升级属于控制命令，不归入普通配置；这类命令可以唤醒设备并要求执行回执
  • 进入 AOV 休眠前必须保证本机已应用配置持久化完成，未生效配置保留为待生效状态
